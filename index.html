Introduction

This documentation site is designed to provide all necessary technical details and guidelines required for interacting with IncludeHealth systems. The site contains comprehensive guides, examples, and references to assist developers and partners in integrating and collaborating effectively with our solutions.

File Transfers

This section provides detailed guidelines about the Secure File Transfer Protocol (sFTP) and CSV data handling for IncludeHealth. It is designed to help software developers understand how to prepare and submit client data, including Personal Health Information (PHI), to IncludeHealth in a secure and compliant manner. It also outlines the data submission format, requirements, retrieval, and error handling procedures.

sFTP Access

Access to the IncludeHealth sFTP servers is provided in separate documentation along with specific credentials for tenant organizations. For additional assistance, please refer to the Help and Support documentation.

Folders

Accessing the sFTP server will grant access to three main folders: input, output, and errors, each serving different functions in the data management process.

Input

This is the target location to place files that will be ingested by the import services.

Output

This is the location where all weekly and monthly files the system generates will be available for retrieval by the tenant organization.

Errors

Should any errors occur during the import process, a file will be created in this folder containing the affected patients' first and last names along with an error message.

Data Submission

Guidelines for software developers on how to prepare and submit client data to IncludeHealth. Please ensure all data conforms to the structure and standards outlined below. The data includes Personal Health Information (PHI) and is protected under a Business Associate Agreement (BAA) with IncludeHealth.

Format

All client data must be submitted in a CSV file with specified required columns. Please review the CSV Example documentation

  • FirstName: First name of the patient. (String)
  • LastName: Last name of the patient. (String)
  • DateOfBirth: Date of birth of the patient, formatted as YYYY-MM-DD. (Date)
  • Email: Email address of the patient. (String)
  • PhoneNumber: Contact phone number of the patient, including only the numeric characters, with the +1 US country code assumed. (Numeric)
  • ExternalPatientId: A unique identifier for the patient. (String or Numeric)
  • PlanId: A unique identifier for the patient's plan, provided by IncludeHealth. This field is an enumeration related to the required assessment for the patient. If a match cannot be made, the patient will not be added, and this will be noted in a "failed results" CSV file. (String)
  • StartDate: Start date for the plan, formatted as YYYY-MM-DD. (Date)
  • ReferringGroup: A Group or category the patient belongs. (String)
  • ReferringClinician: A clinician that requested the referral. (String)
  • TimezoneId: Timezone identifier, consistent with the TZ database format. For a complete list of acceptable timezone identifiers, please refer to the List of TZ Database Time Zones on Wikipedia. (String)
  • BodyPart: The affected body part that the patient is receiving care for. (String)
  • Condition: The condition the patient is receiving care for. (String)
  • SmsMessagingEnabled: Flag to opt the client into receiving sms messaging. (Bool)
  • PlanNotificationType: The type of notifications the client should receive for the plan (Enum: Virtual_PT, Assessment, None).

See the sample template for examples of properly formatted CSV File

Requirements

Every column must be populated with valid data, otherwise, the record will be rejected. Specific formats for dates and other fields are also outlined.

  • All fields are required: Each column must be populated with valid data; otherwise, the record will be rejected.
  • Date Format: All dates must follow the YYYY-MM-DD format.
  • File Format: Data must be submitted in CSV format only.
  • Phone Number Format: Phone numbers must contain only numeric characters. It is assumed that all numbers are prefixed with the +1 US country code.

Data Retrieval

Output files and error files will be placed in the appropriate sFTP folders for retrieval as detailed below.

Output Files

Details on the structured output files provided weekly and monthly, including the types of data they contain and how they are formatted.

Weekly Status

Each week a file will be generated in JSON and xlsx formats and placed in the output folder.

  • Start Date: Represents the start date of the weekly file process in the format "YYYY-MM-DD".
  • End Date: Represents the end date of the weekly file process in the format "YYYY-MM-DD".
  • Summary: Contains a summary of referrals and enrollments.
    • Previous Week Referrals: The number of referrals from the previous week.
    • Total Referrals: The total number of referrals to date.
    • Previous Week Enrollments: The number of enrollments from the previous week.
    • Total Enrollments: The total number of enrollments to date.
    • Currently Enrolled: The number of patients currently enrolled.
  • Details: An array containing detailed information about each patient.
    • ExternalPatientId: A unique identifier for the patient's case provided by the referring organization.
    • Patient Name: The name of the patient in the format Last Name, First Name.
    • ReferringClinician: Clinician who referred the patient. (String)
    • Status and Reason: The current status of the patient (e.g., Actively Engaged) and in some cases, the reason that status was assigned (e.g., Discharged because the patient Met Goals).Standard combinations:
      Status Reason Options
      Pending Initial Evaluation No Reason (zero length string)
      Response - Pending
      Needs Attention
      Evaluation Scheduled No Reason (zero length string)
      Needs Attention
      Actively Engaged No Reason (zero length string)
      Needs Attention
      Discharged Met Goals
      Physician Discharged
      Self Discharged
      Needs Attention
      Closed No Response
      Decline in Health Status
      Lost Benefits
      Self Closed
      Needs Attention
    • Notes: Optional additional information entered by the clinician. This is typically used when patients have a Reason value of Needs Attention, but may be entered for any Status/Reason combination.
    • Weekly Completed Sessions: The number of sessions completed by the patient in the current week.
    • Weekly Active Days: The number of active days for the patient in the current week.
    • Weekly Completed Activities: The number of activities completed by the patient in the current week.
    • Initial Pain: The patient's initial pain score.
    • Initial Pain Date: The date when the initial pain score was recorded in the format YYYY-MM-DD.
    • Current Pain: The patient's current pain score.
    • Current Pain Date: The date when the current pain score was recorded in the format YYYY-MM-DD.
    • Primary Clinician: The name of the primary clinician assigned to the patient in the format Last Name, First Name.
    • Evaluation Date: The date when the patient's evaluation was conducted in the format YYYY-MM-DD.
    • Total Sessions: The total number of sessions completed by the patient to date.
    • Total Active Days: The total number of active days for the patient to date.
    • Total Activities: The total number of activities completed by the patient to date.

This output provides a structured summary of the weekly file process, including overall referral and enrollment statistics, as well as detailed information about each patient's engagement, pain scores, primary clinician, and progress metrics.

The weekly status file is available in JSON and XLSX formats and both are provided

Monthly Status

Summary reports generated on a monthly basis, providing an overarching view of the progress and status changes over the month.

Monthly Consolidated Summary

This report provides information about all current referred patients in the system.

  • ExternalPatientID: A unique identifier for the patient.
  • First Name: The patient's first name.
  • Last Name: The patient's last name.
  • Status: The current status of the patient's case (e.g., active, discharged).
  • Body Part: Specifies the part of the body relevant to the treatment.
  • DOB (Date of Birth): The patient's birth date.
  • Enrollment Date: The date the patient was enrolled in the program.
  • Completed Sessions: The number of sessions the patient has completed.
  • Measures: An array containing detailed information about each assessment outcome.
    • Measurement: Specific measurements related to the body part.
    • Baseline Date: The date initial measurements were taken.
    • Baseline: The initial measurements recorded.
    • Latest Date: The date of the most recent measurement.
    • Latest: The most recent measurements taken.
    • % Change: The percentage change between the baseline and latest measurements.

Monthly Patient Details

This report provides detail information about current referred patients in the system. One file per patient will be created.

  • Patient Information
    • Name: Full name of the patient.
    • ExternalPatientId: Unique identifier assigned to the patient.
    • Email: Email address of the patient.
    • Phone: Contact phone number of the patient.
    • DOB: Date of birth of the patient.
    • Body Part: The specific body part being treated or monitored.
  • Reporting Period
    • Reporting Month: The month for which the report is generated.
    • Episode Start Date: The starting date of the current treatment episode.
    • Episode Status: Current status of the episode (e.g., Active, Inactive).
    • Primary Clinician: Name of the clinician primarily responsible for the patient.
  • Engagement Summary
    • Active Days (Monthly): The number of days the patient accessed services in the month.
    • Active Days (Total): The number of days the patient accessed services.
    • Synchronous Visits (Monthly): The number of real-time visits or interactions within the month.
    • Synchronous Visits (Total): The number of real-time visits or interactions.
    • Completed Sessions (Monthly): The number of therapy or treatment sessions completed in the month.
    • Completed Sessions (Total): The number of therapy or treatment sessions completed.
    • Completed Activities (Monthly): The number of specific activities or tasks completed by the patient in the month.
    • Completed Activities (Total): The number of specific activities or tasks completed by the patient.
  • Surveys
    • Survey Name: Type of survey (e.g., DASH, Owestry, NPS).
    • Classification: The category of the survey (e.g., Initial, Follow-up).
    • Date: Date on which the survey was conducted.
    • Score: The score or outcome of the survey.
    • % From Initial: Percentage change from the initial survey to the current.
  • Functional Tests
    • Test Name: Type of functional test performed (e.g., Lumbar Flexion, Lumbar Sidebending).
    • Classification: The classification of the test (Initial, Follow-up).
    • Date: Date on which the test was conducted.
    • Outcome: The outcome or result of the test in appropriate units.
    • % From Initial: Percentage change from the initial test to the current measurement.

Security and Compliance

Handling PHI: As the data includes PHI, all handling and transmission must comply with the stipulated BAA terms.

Data Transmission: Upload all CSV files to the IncludeHealth sFTP server. Credentials and access details will be provided separately.

Error Handling

Failed Results: If a patient's data cannot be matched to a provided PlanId, it will be flagged in a "failed results" CSV file which will also be available on the sFTP server.

API Access

Technical API Documentation is available here.

Endpoints

There are two API endpoints, one for a "sandbox" environment, and another for production.

  • Sandbox: api-sandbox.includehealth.com
  • Production: api.includehealth.com

API Keys

Keys can be obtained by contacting customer service, and must be appended to the query string of each request as follows.

api.includehealth.com/api/v1/status?code={APIKEY} where "{APIKEY}" is the key assigned to your organization.
